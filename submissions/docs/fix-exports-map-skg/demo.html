Fix: exports map blocks component imports

Issue #53768 — package.json is missing wildcard export entries for components/ and core/, causing ERR_PACKAGE_PATH_NOT_EXPORTED when users try to import individual component stylesheets.

1. What breaks today

Any bundler (Vite, webpack, Rollup, esbuild) or Node.js runtime that respects the exports field will refuse subpaths not declared in it — even if the file physically exists on disk.

// In any project that imports easemotion-css: // ❌ Throws ERR_PACKAGE_PATH_NOT_EXPORTED import 'easemotion-css/components/buttons.css'; import 'easemotion-css/components/cards.css'; import 'easemotion-css/components/modals.css'; // ✅ Works — declared in exports import 'easemotion-css'; // "." import 'easemotion-css/min'; // "./min"
✕ Per-component imports throw resolution errors

2. Current package.json vs proposed fix

// package.json — current (incomplete) "exports": { ".": "./easemotion.css", "./min": "./easemotion.min.css", "./engine": "./easemotion/index.js", "./engine/parser": "./easemotion/engine/parser.js", "./engine/compiler": "./easemotion/engine/compiler.js", "./engine/optimizer": "./easemotion/engine/optimizer.js", "./engine/runtime": "./easemotion/engine/runtime.js", "./core": "./core/animations.css" // ← no components/ or core/*.css entries } // package.json — proposed (add these two lines) "exports": { /* ... existing entries ... */ "./components/*.css": "./components/*.css", "./core/*.css": "./core/*.css" }
ℹ Node.js wildcard subpaths require Node ≥ 12.20 ✓ Project already requires Node ≥ 18 — safe to use
