Bug Fix #13795 — Variable-width font compatibility

⚠️ Bug #13795
Root cause: steps(N) divides the total element width into N equal pixel slices, assuming every character is the same width. On proportional fonts (system-ui, Inter, Roboto), narrow letters like i l r t are 2–3× narrower than wide letters like W M T. Equal pixel slices ≠ equal character steps → cursor visibly misaligns. The original component uses 'Courier New', monospace which protects it — but any font override breaks the animation.

Also fixed: --text-width was hardcoded to 20ch for all instances regardless of actual string length, causing the animation to extend into empty space before reversing on short strings.

How It Works

1. Types character by character using width animation
2. Pauses for 1 second at full width
3. Deletes character by character
4. Pauses for 0.5 seconds at empty
5. Repeats infinitely

Usage — Monospace (original, unchanged)

<div class="ease-typewriter-loop" style="--text-width: 13ch">Hello, World!</div>
<div class="ease-typewriter-loop-fast" style="--text-width: 11ch">Fast typing</div>
<div class="ease-typewriter-loop-slow" style="--text-width: 11ch">Slow typing</div>
<div class="ease-typewriter-loop-delay-1" style="--text-width: 18ch">Staggered</div>

Note: set --text-width to the actual character count of your string.

Usage — Proportional font (new JS fix)

<div class="ease-typewriter-loop-proportional" id="my-tw">Your text here</div>

⚡ Key CSS Concepts

  • 📐 width animation from 0 to ch (character units)
  • 🎯 steps(N, end) for discrete character typing — monospace only
  • 📦 overflow: hidden + white-space: nowrap
  • 💡 border-right animated for blinking cursor
  • 🔄 Chained @keyframes for type → pause → delete → pause cycle
  • 🐛 Bug fix: --text-width now set per-instance to actual string length
  • ✅ New: JS-measured proportional font variant using Range.getBoundingClientRect()
